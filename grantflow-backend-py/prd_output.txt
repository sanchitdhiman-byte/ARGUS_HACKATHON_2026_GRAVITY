1. Overview & Purpose
You are building GrantFlow — an end-to-end Grant Lifecycle Management Platform. The platform
serves a single grant-making organisation that runs multiple grant programmes and manages the full
journey from application intake through to final compliance reporting and fund disbursement.
The platform serves two types of users:
• Internal Staff — Program Officers, Reviewers, Finance Officers — who manage the grant
lifecycle end to end.
• External Applicants — NGOs, institutions — who apply for grants, track their application status,
and submit compliance reports.
What Makes This Platform Distinctive
GrantFlow is not a form-collection tool. It is an intelligent platform where an Agentic AI layer actively
performs meaningful work at critical stages of the lifecycle, reducing manual effort and improving
decision quality. The AI assists in three primary areas:
• When an applicant submits — the platform automatically analyses the application for eligibility
and narrative quality, producing a structured screening report for the Program Officer.
• When a reviewer opens an application — the platform has already read and analysed the full
application, producing a summary, suggested scores, and risk flags before the human reviewer
starts work.
• When a grantee submits a progress report — the platform automatically compares the report
against the approved application, checks financial consistency, and flags anomalies for review.
• When an applicant is filling out an application — the platform offers an AI-guided conversational
mode that walks the applicant through every question instead of presenting a static form.
AI Boundaries
The AI must never take final decisions. Every AI output is a recommendation for a human to
confirm, modify, or override. AI must not send communications directly to applicants, and must
not initiate or approve disbursements. Human accountability is preserved at every decision point.

Module 1 Grant Programme Catalogue
Displays and manages the three pre-configured grant programmes available for applicants and staff to
browse.
Scenarios & Functional Requirements
1.1 Public Grant Discovery
• Any visitor (logged in or not) should be able to view a public-facing page listing all active grant
programmes.
• Each programme listing must display: name, purpose summary, funding range, project duration
range, application deadline, and eligibility highlights.
• Clicking a programme should show a full detail page with all programme attributes.
1.2 Eligibility Pre-Check (No Login Required)
• Any visitor should be able to run a quick eligibility check without registering.
• The pre-check collects: organisation type, project district, and funding amount requested.
• The system evaluates hard eligibility rules only and returns a "Likely Eligible / Likely Not Eligible"
result for each grant programme, with the specific reason for any ineligibility.
1.3 Personalised Eligibility Banner (Logged-in Applicants)
• After login, applicants should see a personalised eligibility banner based on their saved
organisation profile — showing which programmes they are likely eligible for without needing to
re-enter information.
Module 2 Applicant Portal & Application Intake
Enables applicant registration, organisation profiling, document management, and dual-mode
application submission.
Scenarios & Functional Requirements
2.1 Applicant Registration
• New applicants must be able to self-register with: full name, organisation name, email,
password, and phone number.
• Email OTP verification is required before the account is activated.
• After login, applicants complete an Organisation Profile covering: organisation name,
registration number, type, state, annual budget, and contact person. This profile pre-fills fields
across all applications.
2.2 Document Vault
• Applicants must be able to upload standing documents once (Registration Certificate, Audited
Financials, 80G Certificate) to a personal document vault.
• These documents must be automatically attached to new applications and visible for re-use or
replacement.
2.3 Dual-Mode Application
• The manual wizard presents the application with the following sections: Organisation, Project,
Team, Budget, Documents, Review & Submit.
• Each step validates fields on the "Next" click.
• A running total is displayed for the budget section.
• The applicant must confirm a declaration before submitting.
• As an alternative to the wizard, applicants should be able to apply through a conversational AI
interface. The AI interface should have functionality similar to the manual wizard.
AI Chatbot Scope
The AI chatbot is a guided data-collection interface. It asks questions and records what the
applicant provides. It does not write or suggest content for the applicant's answers.
2.4 Document Upload
• Each grant type has a specific document checklist.
2.5 Application Submission & Dashboard
• The Applicant Dashboard shows all applications with: status pill, current workflow stage, and
last updated timestamp.
• Applicants must be able to see a stage timeline indicating which stage their application is at and
what comes next.
• A Messages tab shows all communications from the Program Officer, Applicants can reply via
the portal.
• For active grants, a Reports tab provides the report submission interface.
Module 3 Eligibility Screening
Automatically checks submitted applications against eligibility rules and produces a structured
screening report for Program Officer review.
Scenarios & Functional Requirements
3.1 Hard Rule Checks
• The system must evaluate all eligibility rules for the relevant grant type (as defined in Section 3
of this document for each grant type).
• Each check result (pass or fail) must be individually recorded and displayed in the screening
report.
3.2 Human Validation
• Soft Check 1 — Thematic Alignment: The system must analyse the project title, problem
statement, and proposed solution text. Relevant applications must be soft-flagged for human
validation.
• Soft Check 2 — Narrative Quality: The system must assess the problem statement, proposed
solution, and expected outcomes for: logical coherence between problem → solution →
outcomes; specificity of the beneficiary description; and presence of at least one measurable
outcome indicator. Relevant applications must be soft-flagged for human validation.
• Soft flags are advisory — the Program Officer decides what action to take.
3.3 Screening Report
• The system must generate a structured screening report after all checks are complete.
• The report must be presented to the Program Officer.
3.4 Program Officer Review of Screening
• Program Officers must be able to review the screening report and take one of three actions:
Confirm Eligible (advance to review queue), Override to Ineligible (written reason required), or
Send Clarification Request to applicant.
• Clarification requests: the Officer selects which flag to address, types a question, and the
system sends it to the applicant via the messaging system. The applicant's response is attached
to the application record.
Module 4 Application Review & Scoring
Enables reviewer assignment and structured application review, supported by AI-generated summaries,
scores, and risk flags.
Scenarios & Functional Requirements
4.1 Reviewer Assignment
• Program Officers must be able to assign reviewers to eligible applications from a list of available
staff with the Reviewer role.
• The system must enforce a conflict-of-interest rule: a reviewer whose email domain matches the
applicant's email domain cannot be assigned.
• CDG and ECAG applications require one reviewer; EIG applications require two reviewers.
4.2 AI-Generated Review Package
• Before a reviewer opens an application, the system must automatically generate a Review
Package containing three components:
◦ Application Summary: A structured one-page summary covering who is applying, what they
want to do, where, for how many beneficiaries, for how long, and how much — allowing the
reviewer to understand the full application in approximately two minutes.
◦ AI-Suggested Scores: For each scoring dimension defined in the grant type's rubric, the
system must produce: a suggested score (1–5), a written justification (citing specific text
from the application), and a reference to which section of the application the justification
draws from.
◦ Risk Flags: The system must scan the application for five categories of risk — (1) budget
anomalies where a single line exceeds 60% of the total, (2) unrealistic timelines given the
scope of activities, (3) vague or unmeasurable outcomes, (4) team size appearing
insufficient for the project scope, (5) prior grant history indicating incomplete projects. Each
flag must have a severity level: High / Medium / Low.
• All AI scores must be clearly labelled as "AI Suggested" throughout the interface.
4.3 Reviewer Workspace
• The reviewer workspace must display: application content with document download links, the AI
Review Package (summary, suggested scores, risk flags), and a scoring sheet.
• The scoring sheet must show each rubric dimension with the AI-suggested score. The reviewer
must confirm or override each score.
• If a reviewer overrides an AI-suggested score, a written comment is mandatory.
• Reviewers must be able to highlight text in the application and attach a note to it.
• Scores must auto-save so reviewers can return and continue.
• Once all dimensions are scored, the reviewer submits the review.
4.4 Program Officer Post-Review View
• All reviewed applications must appear in a Program Officer queue showing: composite score,
individual dimension scores, reviewer identity, and risk flags.
• The Officer must be able to sort and filter by score, grant type, reviewer, and date.
• The Officer sets a final decision for each application: Approve / Reject / Waitlist, with a
mandatory written reason.
• The decision must be recorded and automatically trigger the next workflow stage.
Module 5 Grant Award & Agreement Management
Manages award notifications, grant agreement generation, and disbursement schedule configuration.
Scenarios & Functional Requirements
5.1 Award, Rejection & Waitlist Notifications
• On a decision of Approve: the system generates an Award Letter from the defined template and
presents it to the Program Officer for review before sending.
• On a decision of Reject: the system generates a Rejection Letter from the defined template with
the stated reason. The Officer reviews before sending.
• On Waitlist: the system sends a Waitlist Notification. No further action until a slot opens.
• All letters must use the merge field templates defined below.
5.2 Grant Agreement Generation
• Program Officers must be able to generate a Grant Agreement for approved applications by
clicking a "Generate Agreement" action.
• The system must auto-populate the agreement with data from the application and applicant
profile: grantee legal name, registration number, address, grant reference number, programme
name, award date, project title, project description, grant amount, disbursement schedule,
reporting schedule, and any special conditions entered by the Officer.
• The Officer reviews the populated agreement and sends it to the grantee as a PDF.
• The grantee acknowledges the agreement via a link in their portal — the acknowledgement is
recorded with a timestamp.
5.3 Disbursement Schedule Configuration
• During agreement generation, the Program Officer configures disbursement tranches.
Supported types:
◦ Inception Tranche — released on agreement acknowledgement (used for CDG and ECAG).
◦ Mid-Project Tranche — released when the 6-month report is approved by the Program
Officer.
◦ Final Tranche — released when the final report is approved and accepted.
◦ Milestone-based Tranche — released on approval of a specific deliverable (used for EIG).
• Each tranche must have: a label, an amount (INR), a trigger condition, and a status (Pending /
Ready / Disbursed).
• Finance Officers must be able to add and view bank details for the grantee (account number,
IFSC, beneficiary name). This information must be accessible only to the Finance Officer role.

6.2 AI Compliance Analysis
• When a report is submitted, the system must automatically analyse it and prepare a compliance
analysis for the Program Officer — no manual trigger required.
• The content analysis must evaluate: whether activities described are consistent with what was
approved in the application; whether beneficiary progress is on track relative to the project
timeline; whether the narrative acknowledges any significant deviation from the plan; and
whether the report appears to be balanced (i.e., acknowledges challenges rather than being
entirely positive).
• The financial analysis must verify: arithmetic consistency between period expenditure,
cumulative expenditure, and line-item totals; any budget line where cumulative spend deviates
more than 10% from the approved budget; whether a variance explanation exists for each
deviated line; and underspend alerting — if cumulative spend is below 30% of total budget at
50% or more of the project timeline.
• The analysis must produce an overall content quality rating (Satisfactory / Needs Clarification /
Concerns Found), a list of specific flags with field references, and a recommended action.
6.3 Program Officer Report Review

• Compliance Action flags must have a severity level: Warning / Disbursement Hold. A

6.4 Fund Utilisation Dashboard
